## 판정 기준의 의미

| 상태 | 정의 | 엄격 분류 |
|------|------|---------|
| `pass` | 객관적 `verify`가 기대값과 일치 (output_contains·file_exists 등 하드 체크) | **Pass** |
| `qa_fallback` | 하드 체크 실패, LLM이 *의미 판정*으로 "의도 충족"이라 판단 | **Fail** |
| `fail` | 하드 체크 실패 + LLM도 부정 | **Fail** |
| `no_execution` | 실행 자체 실패 (인프라·권한 문제) | **Fail** |
| `untested` | 아직 실행 안 됨 | 평가 대상 외 |

## 관찰 — 엄격 기준에서의 교훈

- **Pass 35.8%** 는 *절대값으로는 낮다*. 초기 기대치 대비 강한 격차.
- *LLM 판정에 기대지 않고* 개선해야 할 지점들:
  - **qa_fallback 835건**: 대부분 **verify.expect 설정이 느슨/과엄격**한 경우가 많다. 주차별 재작성 필요.
  - **fail 516건**: 명령·스크립트 오류 또는 인프라 미지원 — 수정·대체 필요.
  - **no_execution 7건**: 환경 문제 — 인프라 수리 필요.
- **상위 3개 과정**(secops 80% · soc-adv 73% · soc 68%)은 *verify 설계가 잘 된 과정*. 이들의 구조가 다른 과정의 개선 참고.
- **하위 과정들**(autonomous 3%·autonomous-systems 8%)은 *실제로 실행이 거의 안 되는 상태* — Untested 포함 고려 시 사실상 미점검.
