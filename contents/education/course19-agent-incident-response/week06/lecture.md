# Part 1: 회피의 3축 (30분)

## 1.1 공격 페이로드는 어떻게 변할 수 있나

| 축 | 기법 | 예시 |
|----|------|------|
| **인코딩** | URL/HTML/Base64/Unicode 이중 인코딩 | `UNION SELECT` → `%55NION%20SELECT` |
| **분할** | 공격을 다수 요청으로 나누기 | SQLi를 2개 요청으로 분할 |
| **타이밍** | 요청 간 지연 주입 | 정상 트래픽 속도로 위장 |

에이전트는 이 3축을 **자동 조합**해 시도한다.

### 1.1.1 인코딩 축 — 실제 변형 예시

SQLi 페이로드 `' OR 1=1--`를 예로 들면, 에이전트가 자동으로 생성하는 변형.

| 기법 | 결과 |
|------|------|
| 원본 | `' OR 1=1--` |
| URL 인코딩 1x | `%27%20OR%201%3D1--` |
| URL 인코딩 2x | `%2527%2520OR%25201%253D1--` |
| Unicode full-width | `' ＯＲ 1＝1--` |
| HTML 엔티티 | `&#39; OR 1=1--` |
| Hex | `%30x27 OR 1=1--` (대부분 해독기에서 처리) |
| 공백 대체 | `'/**/OR/**/1=1--` |
| 주석 분할 | `'/*foo*/OR/*bar*/1=1--` |
| Null byte | `' OR 1=1--%00` |
| 대소문자 혼합 | `' oR 1=1--` (경우에 따라 WAF 우회) |

에이전트는 이 변형을 *한 턴*에 10~30개 생성할 수 있다. ModSec 기본 CRS가 각각을 *모두* 차단할지는 **룰 품질**에 달렸다.

### 1.1.2 분할 축 — 공격 한 건을 여러 요청으로

분할은 *단일 페이로드가 시그니처에 안 걸리게* 하는 기법. 예:

```
원본 (1개 요청):
POST /search body={"q": "' UNION SELECT user,pass FROM users--"}

분할 (2개 요청):
POST /search body={"q": "' UNION"}
POST /search body={"q": "SELECT user,pass FROM users--"}
```

서버가 *세션 상태*를 유지하면 공격이 성립한다. 대부분의 WAF는 *단일 요청* 기반 검사라 분할에 취약하다.

### 1.1.3 타이밍 축 — 정상 속도로 위장

에이전트는 요청 간격을 *정상 사용자*처럼 조정할 수 있다 (지시 시). 방어 관점에서는:

- 요청 *속도*는 정상 범위에 있어도
- 요청 *내용*(페이로드·경로)은 비정상
- → *내용 지표*에 집중해야 함 (속도 지표는 우회됨)

## 1.2 변형 생성의 경제학
- 사람이 변형 1개 만드는 데 수 분
- 에이전트가 변형 20개 만드는 데 수 초
- *룰 1장이 20개 변형을 모두 커버할 가능성은 낮음*

## 1.3 방어의 의미 — "룰 쌓기"의 함정
룰을 계속 추가하면 **오탐 증가 + 성능 저하**. 에이전트는 그 사이를 파고든다. 그래서 방어는 **"룰 수"가 아니라 "세션의 변형 진화 탐지"**로 이동한다.

# Part 2: 에이전트의 역탐지 (30분)

## 2.1 역탐지란
에이전트가 **방어 측이 무엇을 보고 있는지**를 추정하는 행위.
- 403 응답 본문에 *ModSec rule id* 노출되면 → 그 룰을 직접 우회
- 응답 지연 패턴이 규칙적이면 → tar-pit 의심, 세션 폐기
- 갑작스레 정상 응답이 가짜 데이터로 바뀌면 → honeypot 의심

## 2.2 방어 측의 *일관성 유지* 원칙
- 에러 응답 **축약**하고 **통일**
- 탐지 이후 응답을 **급격히 바꾸지 않음**(기만은 *전략적 시점*에)
- Rate-limit 임계값도 **균일하게 보이도록** 난수화

### 2.2.1 역탐지 대응 — "단서 최소화" 10개 체크리스트

방어 측에서 에이전트에게 *단서를 주지 않기* 위한 점검.

- [ ] 응답에 **X-Powered-By·Server 헤더** 포함 여부
- [ ] 404/500의 **상세 메시지** 축약
- [ ] WAF의 **룰 ID·모듈명** 노출 금지
- [ ] 로그인 실패 응답의 "사용자 없음" vs "비밀번호 틀림" **통합**
- [ ] Rate-limit 당시 응답 시간의 **일관성** (정상과 차이 최소)
- [ ] 쿠키 세팅·삭제 **자동 응답** (결함 노출 방지)
- [ ] 세션 토큰의 **난수성·길이 동일**
- [ ] CAPTCHA 발동 조건 *난수화* (예측 차단)
- [ ] 허니토큰이 *정상 데이터와 섞여*(분리 로깅이 이를 구분)
- [ ] CORS 응답이 *일관*(prod/dev 혼용 금지)

이 10가지가 조직 환경 점검의 1차 체크리스트다.

### 2.2.2 역탐지 *유발* 전략 (가끔은 유용)

반대로 에이전트가 *특정 결론*을 내리도록 *의도적 단서*를 주는 것이 이득인 경우도 있다.

- **가짜 WAF 룰 ID 노출**: 공격자가 *존재하지 않는 룰*을 우회하려 시간 낭비
- **가짜 버전 힌트**: 오래된 취약 버전처럼 보이게 해 공격 경로 제한
- **가짜 장애 시늉**: 특정 공격 시 서버 오류처럼 보이게 해 공격 철회 유도

이 기법은 *고급 기만*이며, w10에서 다룬다. 초보 방어자는 *역탐지 대응*(단서 최소화)을 먼저 완성한 뒤 *역탐지 유발*로 넘어간다.

# Part 4: 롤링 탐지 설계 (40분)

## 4.1 "변형 진화" 지표 (세션 단위)
- 페이로드 문자열의 Levenshtein 거리 분포: 시간 순으로 **점진적 증가**인지
- 인코딩 종류의 **다양성** (URL vs Base64 vs Unicode 등)
- 403/200 사이의 **전이 속도**

## 4.2 Bastion이 할 수 있는 것
- 세션의 의심 점수가 임계 이상 → *응답 지연 주입*(w10)
- Suricata에 **자동으로 파생 룰 생성**(부모 룰 매칭 시 변형 자동 파생)
- 매 N분마다 *룰 히트 분포*를 모델에 질의해 새 룰 후보 제안

## 4.3 구현 스니펫 아이디어
```python
# 세션의 변형 진화 탐지 (개념 코드)
def evolving_payloads(session_requests):
    bodies = [r.body for r in session_requests]
    from difflib import SequenceMatcher
    diffs = [1-SequenceMatcher(None, a, b).ratio() for a,b in zip(bodies, bodies[1:])]
    return sum(1 for d in diffs if 0.05 < d < 0.4) >= 5  # 점진적 변화 5+회
```

# Part 5: 정적 룰의 한계와 세션 상관관계 (30분)

## 5.1 핵심 주장
- 에이전트 앞에서는 **한 요청 한 룰**의 모델이 무너진다
- **세션 = 관찰 단위**로 바뀌어야 한다
- SIEM 쿼리가 *요청*이 아닌 *세션*을 묶을 수 있어야 함

## 5.2 실무 권고 (세션 식별의 어려움)
- 세션 쿠키 기반은 에이전트가 회피 가능
- *IP + User-Agent + TLS 지문*의 앙상블이 현실적
- BunkerWeb이 제공하는 `req_id`와 Suricata flow_id의 상관관계

### 5.2.1 TLS 지문(JA3/JA4) 실습 — 세션 식별의 강력한 축

TLS ClientHello 패킷에서 *암호 스위트·확장·순서*가 클라이언트별로 다르다. 이를 해시한 것이 JA3(구)·JA4(신). Suricata는 이 값을 이벤트에 기록한다.

```bash
# secu의 Suricata eve.json에서 JA3 확인
sudo jq 'select(.tls.ja3) | {ts:.timestamp, src:.src_ip, ja3:.tls.ja3.hash, ua:.tls.ja3s.hash}' \
  /var/log/suricata/eve.json | tail -20
```

같은 JA3를 가진 요청은 *동일 클라이언트 프로세스*로 간주. IP가 달라도 JA3가 같으면 *같은 공격 도구*.

### 5.2.2 세션 클러스터링의 실무 — Python

```python
def cluster_sessions(events):
    """IP·UA·JA3 앙상블로 세션 묶기"""
    from collections import defaultdict
    clusters = defaultdict(list)
    for e in events:
        key = (e.ja3 or e.ja4, e.user_agent_class, e.src_ip_24)
        clusters[key].append(e)
    # 3개 이상 멤버를 가진 클러스터만 "의미 있는 세션"
    return {k: v for k, v in clusters.items() if len(v) >= 3}
```

IP/24 단위로 느슨히 묶어 IP 로테이션도 어느 정도 흡수.

### 5.2.3 BunkerWeb의 `req_id` 활용

BunkerWeb은 각 요청에 UUID `X-Request-ID`를 붙인다. 이 ID가 *Apache access log*와 *ModSec audit log*를 연결한다.

```bash
# 같은 req_id로 두 로그를 교차 조회
grep -E "X-Request-ID: abc-123" /var/log/apache2/access.log
grep -E "abc-123" /var/log/bunkerweb/modsec_audit.log
```

학생은 자신의 세션에서 한 `req_id`를 골라 *양쪽 로그*에서 흔적을 추적해 본다.

## 부록 A. CRS 룰 카테고리 개요

|------|---------|
| 901xxx | 초기화·공통 |

| 913xxx | 스캐너 탐지 |
| 920xxx | 프로토콜 검증 |
| 930xxx | 경로 이동(Path Traversal) |
| 932xxx | 명령 주입 (OS Command Injection) |

에이전트가 "942xxx 룰을 우회하려" 할 때 *우리는 다른 범위*(930·932)로도 추가 매칭을 고려해야 한다.

## 부록 B. "우회 성공 직전"의 공통 징후

실습 통계로 본 *우회가 곧 이루어질* 위험 신호.

- 403 비율이 **급격히 감소**(100% → 80% 이하)
- 페이로드 크기가 **점점 작아짐**(분할 중)
- 응답 본문 크기가 **정상화**(WAF 차단 페이지에서 원본 앱 응답으로)
- 에이전트의 요청이 *서로 다른 엔드포인트*에 분산

이 네 신호가 *동시 관찰*되면 "우회 임박" 판정. Bastion이 이 조합에 대해 **강경 대응**(tar-pit + 즉시 차단)으로 전환해야 한다.
