# Part 1: 정찰의 재정의 (30분)

## 1.1 전통적 정찰 루브릭 — 사람 기준

| 단계 | 활동 | 사람 팀 소요 |
|------|------|--------------|
| 외부 자산 식별 | DNS·WHOIS·서브도메인 | 반나절 |
| 서비스 지문 | nmap/banner/HTTP 헤더 | 수 시간 |
| 기술 스택 추정 | 프레임워크·버전·경로 | 반나절 |
| 알려진 취약점 매핑 | CVE·Exploit-DB 조회 | 수 시간 |
| 공격면 종합 | 요약 문서 | 하루 |

총: **2~5일**

### 1.1.1 전통 정찰의 세부 구조 — 비교 기준선

왜 사람 팀에게 2~5일이 필요한가. 각 단계가 *기다림*으로 가득한 구조이기 때문이다.

- **DNS 조회**: 응답 1초, 분석 5~10분. → 수십 도메인 반복하면 반나절.
- **서브도메인 열거**: DNS 부하 우려로 *속도를 조절*. 사람이 쿼리 주기를 관리.
- **포트 스캔**: 네트워크 규모별 수 분~수 시간. 대역폭 절약 옵션(`-T2`) 필요.
- **배너·헤더 분석**: 결과를 *엑셀·노트*로 옮기며 수작업 정리.
- **CVE 조회**: 각 버전에 해당하는 CVE를 *외부 DB*(NVD·Exploit-DB)에서 하나씩 확인.
- **문서화**: 결과를 공격 계획용 요약으로 *재작성*. 수 시간.

이 구조에서 **"단계 간 전환"**이 가장 큰 비용이다. 단계마다 사람이 *결과를 읽고·판단하고·다음 명령을 구성*한다. 이 과정이 정찰의 80% 이상을 차지한다.

### 1.1.2 에이전트가 건너뛰는 것 — 전환 비용의 소멸

에이전트의 정찰이 빠른 근본 이유는 *단계 소요 시간*이 줄어서가 아니라, **단계 사이의 사람-루프가 제거**되기 때문이다.

```
[전통 정찰]  DNS → 🧍 분석 → 스캔 → 🧍 분석 → 배너 → 🧍 분석 → CVE → 🧍 보고
             ←— 🧍 판단 ——→  ←— 🧍 판단 ——→  ←—— 🧍 판단 ——→

[에이전트]  DNS → 스캔 → 배너 → CVE → 보고
            └────── LLM 내부 turn 내 판단 ──────┘
```

이 차이가 "수십 배 가속"의 비밀이다. 따라서 방어가 *개별 단계*에 침투해 속도를 늦춰도, 에이전트는 *단계 사이의 여유*가 없으므로 같은 효과를 얻기 어렵다. 방어는 *단계 자체의 실패 유도*로 전환해야 한다.

## 1.2 에이전트 정찰 — 관측값 (본 실습 인프라 내부 기준)

| 단계 | 관측 시간 | 특이점 |
|------|----------|--------|
| 외부 자산 식별 | ≤ 1분 | *"주어진 대상만 본다"* 정책 덕에 좁음 |
| 서비스 지문 | 1~2분 | `curl -I` + 로봇/사이트맵 병렬 |
| 스택 추정 | 1분 | HTML·헤더의 *암시* 추론 |
| 알려진 취약점 매핑 | 1~2분 | 내부 지식 + RAG 인용 |
| 종합 보고 | 1분 | 한 메시지 안에 Markdown 요약 |

총: **5~8분**. 사람 대비 **수십 배의 압축**.

## 1.3 왜 이렇게 빠른가 — 세 가지 이유

1. **대기 시간 거의 없음**: 사람이 다음 명령을 생각하는 시간이 0에 수렴.
2. **병렬 도구 호출**: `Bash`, `WebFetch`, `Grep`을 *한 계획 안에서* 엮는다.
3. **추론 내장**: 일반 공격 지식·CVE·프레임워크 구조가 **모델 가중치에 이미 있음**.

이 세 요인은 *모두* 모델/하네스 자체 특성이다. 방어 측이 원천 차단할 수 없다. **늦어지게 만드는 것**만 가능하다(w10의 tar-pit·deception).

### 1.3.1 "추론 내장"의 구체 — 에이전트가 *이미 아는* 것들

에이전트의 정찰 속도는 *정보 검색*보다 *이미 아는 지식의 재구성*에서 온다. 대표 예:

- **OWASP Top 10** 전체를 "알고 있음" → URL 하나를 보면 어떤 카테고리 공격이 가능한지 즉시 추정
- **JuiceShop·DVWA·bWAPP** 등 유명 실습 환경의 챌린지 구조를 *외우고 있음*
- **공통 CVE의 PoC 문자열** (예: CVE-2021-44228 Log4Shell의 `${jndi:...}`) 직접 생성 가능
- **프레임워크 경로**: `/actuator/*` (Spring Boot), `/.env` (Laravel), `/_next/` (Next.js) 등의 관례
- **포트↔서비스 맵핑**: 22 SSH, 443 HTTPS, 6379 Redis, 9200 Elasticsearch, 27017 MongoDB 등 즉각 연상

이 내장 지식은 *정찰의 첫 1~2분*을 **외부 조회 없이** 완수하게 한다. 방어 관점에서는 이 단계가 *가장 조용*하므로 탐지가 가장 어렵다.

### 1.3.2 에이전트 정찰이 *느려지는* 경우

항상 빠르지는 않다. 다음 조건에서는 사람 대비 이점이 작아진다.

- **자체 프레임워크·내부 DSL**: 모델이 학습 데이터에서 본 적이 없는 기술. 정찰 시 *구조 추론*에 실제 시간이 필요.
- **비표준 포트**: 표준 매핑이 안 통해 각 서비스의 *행동 관찰*로 추정해야 함.
- **고도의 인증 경계**: 비인증 엔드포인트가 *없는* 환경. 정찰 자체가 *의미 있는 응답을 못 얻음*.
- **심층 IPS/WAF**: 요청 자체를 *아예 차단*. 에이전트도 더 이상 정보를 얻지 못함.

방어 설계의 힌트: 이 네 조건 중 *비표준 포트*는 obscurity 원칙에 반해 권장되지 않지만, 나머지 세 조건은 *본질적 방어*로 유효하다.

### 1.3.3 에이전트 정찰의 *철회 비용*

에이전트가 *틀린 가설*로 정찰을 돌렸을 때 되돌리는 비용은 어떤가.

- 사람: 수 시간 낭비 후 "아, 이 방향이 아니었구나" 인지 → 재시작
- 에이전트: 수 분 낭비 후 *다음 턴*에서 "기술 스택 추정이 틀렸음" 자가 인지 → 재시도

철회 비용도 에이전트가 훨씬 작다. 이 점이 방어 측에는 나쁜 소식이다 — *잘못된 단서*를 심어도 에이전트는 빠르게 회복한다. 허니팟·기만의 설계도 이 *회복 시간* 안에 *다음 단서*를 준비해야 한다.

## 2.1 MITRE ATT&CK — 정찰 단계 기법 매핑

| 활동 | Tactic | Technique |
|------|--------|-----------|
| 공개 URL 탐색 | TA0043 Recon | T1595.002 (Vulnerability Scanning) |

## 2.2 이번 주에 수집할 "정찰 신호 원자" 5종

| 신호 | 정의 | 저장 경로(권장) |
|------|------|-----------------|
| *ReqBurst* | 동일 소스에서 2분 내 **N개 이상 경로** 요청 | secu — tcpdump 파생 |
| *FailSeq* | 4xx 응답 후 3초 내 변형 요청 시퀀스 | web access.log |
| *FingerQuery* | 헤더·버전 추출 목적 요청 패턴 | web access.log |
| *ToolMix* | 같은 소스에서 curl+브라우저+자동 스캐너 섞임 | secu pcap |
| *SessionSilence* | 쿠키 사용·세션 지속 없이 요청만 연속 | web access.log |

이 5개는 w5 탐지 룰의 원료가 된다.

### 2.2.1 각 신호의 구체 측정법

**ReqBurst**
- 측정: 슬라이딩 윈도우 2분, 동일 `src_ip` 기준 고유 경로 수
- 임계: 10개 이상을 1차, 20개 이상을 강경보
- 확인: `tail -1000 /var/log/apache2/access.log | awk '{print $1, $7}' | sort | uniq | awk '{print $1}' | sort | uniq -c`

**FailSeq**
- 측정: 4xx 응답 → 다음 요청까지 시간 → 본문 유사도
- 임계: 3초 내 재시도 && 본문 유사도 0.5~0.95(부분 변형) 5회 이상
- 확인: pcap의 HTTP 요청을 시간순 정렬 후 `diff` 기반 유사도 계산

**FingerQuery**
- 측정: 정보 수집 목적이 명확한 요청 (`/server-status`, `/.git/`, `/actuator/info`, `/package.json`)
- 임계: 1분 내 5개 이상 다른 경로
- 확인: URI 화이트리스트를 `jq`로 필터링

**ToolMix**
- 측정: 같은 `src_ip`에서 User-Agent 다수 섞임
- 임계: 5분 내 UA 3종 이상 (CURL + Mozilla + scanner)
- 확인: `awk -F\" '{print $6}' access.log | sort -u`

**SessionSilence**
- 측정: 요청은 있지만 `Cookie` 헤더 없음 또는 매번 새 쿠키
- 임계: 10요청 이상 동일 src에서 쿠키 부재 또는 세션 불연속
- 확인: `awk '{print $1, $12}' access.log` 로 쿠키 관찰

### 2.2.2 "정찰 신호 × 합법 패턴" 혼동 예방 가이드

각 신호가 어떤 *합법 행동*과 혼동될 수 있는지 정리해, 오탐 관리의 베이스라인을 만든다.

| 신호 | 혼동 위험 | 구분 힌트 |
|------|----------|-----------|
| ReqBurst | 검색 크롤러·사내 모니터링 | User-Agent·허용 목록 |
| FailSeq | 사용자 반복 로그인 실패 | 동일 `email` vs 변형 |
| FingerQuery | 관리자 헬스체크 | 접근 IP 허용 목록 |
| ToolMix | 개발자 디버깅 | 시간대·소스 네트워크 |
| SessionSilence | API 클라이언트 | API 키 존재 여부 |

이 표를 *조직 맞춤*으로 확장한 것이 w5 탐지 룰의 **기초 베이스라인**이다.

### 2.2.3 ATT&CK 기법별 *신호↔기법* 역방향 매핑

신호에서 ATT&CK 기법으로 *역으로* 매핑할 수 있어야 IR 보고서가 가능하다.

| 신호 | 해당 Technique | 해당 Tactic |
|------|---------------|-------------|
| ReqBurst | T1595.002 (Vulnerability Scanning) | TA0043 |
| FingerQuery | T1592.002 (Software) / T1590 (Gather Victim Network Info) | TA0043 |
| FailSeq | T1078 (Valid Accounts) 시도 | TA0006 Credential Access |
| ToolMix | T1588.002 (Obtain Tool) | TA0042 Resource Dev |
| SessionSilence | T1592 (Gather Victim Host Info) 패턴 | TA0043 |

본 매핑은 w8 중간평가, w13 보고서에서 *그대로 사용*된다.

## 3.4 Post-run 통계 수집 스크립트 — 스니펫

```bash
# secu에서 패킷 캡처 후 요청 간격(CoV) 분석
tcpdump -i any -nn -w /tmp/recon.pcap 'host 10.20.30.80' &
# ... (실험 실행) ...
# 종료 후
tshark -r /tmp/recon.pcap -Y 'http.request' -T fields \
  -e frame.time_epoch -e ip.src -e http.request.uri \
  > /tmp/recon.tsv
python3 - <<'PY'

## 3.5 실험 목표 지표 (학생 기준 달성)

### 3.5.1 실험 운영 순서 (10분 단위)

```
[T-5] 준비: 강사가 안전 수칙 재확인, pcap 캡처 시작

각 학생은 자신의 세션별로 다른 결과를 얻는다 (모델 출력의 확률성). 그룹 전체 결과를 비교하면 *에이전트 정찰의 변동성*을 체감할 수 있다.

### 3.5.2 관찰된 공통 패턴 (이전 코호트 통계)

여러 코호트의 실험 결과 통계 (대략):

| 지표 | 중간값 | 5~95% 범위 |
|------|--------|-----------|
| 총 요청 수 | 40 | 20~120 |
| 고유 경로 수 | 18 | 8~45 |
| IAT 평균 | 0.8s | 0.4~2.0s |
| IAT CoV | 0.22 | 0.12~0.45 |
| 4xx 비율 | 35% | 15~60% |
| 정찰 종료까지 시간 | 4.5분 | 2~9분 |

이 범위가 *에이전트 지문*의 *정상 분포*다. 여기를 벗어나는 세션은 *세분화된 지문*(이번 주 4.4절 참고) 단서를 조사해야 한다.

### 3.5.3 에이전트의 "예상 외 행동" 기록

학생이 실험에서 가끔 보는 *예상 외* 행동을 기록하면, Purple Round에 귀한 자료가 된다.

- **에이전트가 *자기 제한을 걷어내는* 순간** (예: "합법 교육 환경이라 하셨으니…")
- **에이전트가 *허위 성공*을 주장하는 순간** (실제로는 실패한 공격을 성공으로 보고)
- **에이전트가 *엉뚱한 경로*에 빠지는 순간** (robots.txt에 적힌 가짜 경로를 심각하게 추적)
- **에이전트가 *자발적으로 중단*하는 순간** ("이 이상은 도덕적으로 진행할 수 없음")

각 사례는 *방어 설계*의 힌트가 된다. 특히 세 번째는 *허니팟의 효과* 증거다.

### 4.4.1 해석 코멘트 작성법

각 지표 옆의 *해석* 열은 단지 값을 다시 쓰는 곳이 아니다. 다음 네 요소를 짧게 언급한다.

1. **방향**: 정상 대비 크거나 작음
2. **크기**: 얼마나 벗어났나 (z-score 또는 %)
3. **의미**: 어떤 공격자 행동과 일치
4. **제안**: 다음 관찰 또는 대응

예:
```
지표: IAT CoV = 0.18
해석: 정상 평균 대비 작음 (약 -1.5σ), 기계적 호출 지문, 
      → Suricata 세션 상관 룰에 임계 0.3 설정 제안
```

### 4.4.2 불확실성 표현

관찰만으로 *100% 에이전트 확정*은 어렵다. 보고서에 불확실성을 반드시 명시한다.

- *"추정"*: 복수 지표가 지문을 시사함
- *"유력"*: 주요 3 지표 이상 일치
- *"확정"*: 허니토큰 접촉 + 세션 상관관계

각 단계에 따라 **대응 강도**가 달라진다. 추정은 *관찰 강화*만, 유력은 *레이트리밋*, 확정은 *차단·보고*.

### 4.4.3 Wazuh Discover 쿼리 예시

실제로 대시보드에서 다음 KQL을 시도해 보면 정찰 패턴이 드러난다.

```
rule.groups: "web" AND data.http.status_code: [400 TO 499] AND rule.level >= 3
# 최근 30분으로 필터링하면 학생 세션의 4xx 급증이 보임

rule.groups: "suricata" AND rule.level >= 5
# Suricata의 고급 경보만

agent.name: "web" AND data.url: /robots.txt
# 로봇 파일 접근 기록
```

이 쿼리를 저장해 두면 w5·w8의 *표준 분석 쿼리*로 사용할 수 있다.

# Part 5: 탐지의 3가지 후보 전략 (30분)

## 5.1 전략 A — 행위 임계값

IAT·Burst·경로 다양성을 조합한 *행위 점수*. 단점: 합법 스캐너/벤더와 구분 어려움.

## 5.2 전략 B — 컨텍스트 일관성

세션·쿠키·리퍼러의 일관성을 본다. 사람 세션은 연속적, 에이전트는 이산적.

## 5.3 전략 C — 능동 기만(후속 주차 예고)

정찰 경로에 *허니 자산*을 심어 **접촉만으로도** 고신뢰 탐지. w10 주제.

## 5.4 그룹 결정

각 그룹은 A/B/C 중 하나를 골라 w5의 SIGMA 룰로 **한 문장 초안**을 만든다.

### 5.4.1 세 전략의 *시너지*

A/B/C는 *경쟁*이 아니라 *레이어*다. 실제 방어는 셋을 *함께* 건다.

- A(행위 임계)가 1차 필터 — 빠르고 저렴
- B(컨텍스트 일관성)가 2차 — 오탐 제거
- C(능동 기만)가 3차 — 고신뢰 확인

1차만 쓰면 오탐이 많고, 3차만 쓰면 탐지율이 낮다. 셋의 조합에서 *정밀도·재현율의 균형*이 나온다.

### 5.4.2 전략 조합의 실패 모드

- **A만 세게 걸었을 때**: 정상 트래픽 차단으로 사업 영향 → 관리자 분노
- **B만 세게 걸었을 때**: 에이전트가 *일관된 세션* 흉내를 내면 통과
- **C만 세게 걸었을 때**: 허니 자산을 넘어서는 공격은 무시됨

각 전략의 *단일 실패*를 의식하며 조합해야 한다.

### 5.4.3 다음 주(w4) 연결

본 주차의 정찰 신호는 **w4의 익스플로잇 자동화**에 *직접* 이어진다. 에이전트는 정찰 결과를 *즉시* 익스플로잇 도구 생성으로 연결한다. 따라서 정찰과 익스플로잇 간 *시간·소스 상관*이 **세션 단위 판정의 강력한 단서**가 된다.

w4의 예고: 동일 src가 w3 정찰 패턴 → 1~2분 내 w4 페이로드 전환하면 *정찰→악용 전환*으로 높은 신뢰 판정 가능.
