# Week 03: 초지능 정찰 — 사람 1주가 분 단위로 압축된다

## 이번 주의 위치
w1에서 직시한 속도의 붕괴는 *정찰 단계*에서 가장 먼저 드러난다. 사람 팀이 1~2주 동안 분업으로 수행하던 외부 정찰·기술 스택 추정·알려진 취약점 매핑이 한 에이전트 세션에서 **8분 이내**에 끝난다. 이번 주는 그 과정을 학생이 **스스로** Claude Code에 의뢰해 보고, **방어자 관점**에서 그 과정을 관찰·수집·평가한다. w5의 탐지·w10의 기만은 이번 주에 수집한 신호를 원료로 삼는다.

## 학습 목표
- 에이전트 기반 정찰의 **단계적 내러티브**를 직접 관찰·기록한다
- 정찰 단계의 각 활동을 **MITRE ATT&CK Reconnaissance (TA0043)** 및 후속 Resource Development(TA0042)와 매핑한다
- 사람 팀의 정찰과 에이전트 정찰의 **행위 지문 차이**를 정량 지표로 작성한다
- 방어자 관점에서 정찰을 *탐지하기 어려운 이유*를 분석하고, *늦어지게 만드는 방법*의 후보를 세 가지 이상 제시한다
- 본 주차에 수집된 패킷·로그·세션 기록을 w4·w5 입력 자산으로 정리한다

## 전제 조건

- Claude Code CLI 접근 (강사가 제공하는 공격자 워크스테이션 세션)

# Part 1: 정찰의 재정의 (30분)

## 1.1 전통적 정찰 루브릭 — 사람 기준

| 단계 | 활동 | 사람 팀 소요 |
|------|------|--------------|
| 외부 자산 식별 | DNS·WHOIS·서브도메인 | 반나절 |
| 서비스 지문 | nmap/banner/HTTP 헤더 | 수 시간 |
| 기술 스택 추정 | 프레임워크·버전·경로 | 반나절 |
| 알려진 취약점 매핑 | CVE·Exploit-DB 조회 | 수 시간 |
| 공격면 종합 | 요약 문서 | 하루 |

총: **2~5일**

## 1.2 에이전트 정찰 — 관측값 (본 실습 인프라 내부 기준)

| 단계 | 관측 시간 | 특이점 |
|------|----------|--------|
| 외부 자산 식별 | ≤ 1분 | *"주어진 대상만 본다"* 정책 덕에 좁음 |
| 서비스 지문 | 1~2분 | `curl -I` + 로봇/사이트맵 병렬 |
| 스택 추정 | 1분 | HTML·헤더의 *암시* 추론 |
| 알려진 취약점 매핑 | 1~2분 | 내부 지식 + RAG 인용 |
| 종합 보고 | 1분 | 한 메시지 안에 Markdown 요약 |

총: **5~8분**. 사람 대비 **수십 배의 압축**.

## 1.3 왜 이렇게 빠른가 — 세 가지 이유

1. **대기 시간 거의 없음**: 사람이 다음 명령을 생각하는 시간이 0에 수렴.
2. **병렬 도구 호출**: `Bash`, `WebFetch`, `Grep`을 *한 계획 안에서* 엮는다.
3. **추론 내장**: 일반 공격 지식·CVE·프레임워크 구조가 **모델 가중치에 이미 있음**.

이 세 요인은 *모두* 모델/하네스 자체 특성이다. 방어 측이 원천 차단할 수 없다. **늦어지게 만드는 것**만 가능하다(w10의 tar-pit·deception).

## 2.1 MITRE ATT&CK — 정찰 단계 기법 매핑

| 활동 | Tactic | Technique |
|------|--------|-----------|

| 공격 도구 내부 생성 | TA0042 Resource Dev | T1588.001 (Malware) *acquire* 대신 *generate* |

## 2.2 이번 주에 수집할 "정찰 신호 원자" 5종

| 신호 | 정의 | 저장 경로(권장) |
|------|------|-----------------|
| *ReqBurst* | 동일 소스에서 2분 내 **N개 이상 경로** 요청 | secu — tcpdump 파생 |
| *FailSeq* | 4xx 응답 후 3초 내 변형 요청 시퀀스 | web access.log |
| *FingerQuery* | 헤더·버전 추출 목적 요청 패턴 | web access.log |
| *ToolMix* | 같은 소스에서 curl+브라우저+자동 스캐너 섞임 | secu pcap |
| *SessionSilence* | 쿠키 사용·세션 지속 없이 요청만 연속 | web access.log |

이 5개는 w5 탐지 룰의 원료가 된다.

## 3.4 Post-run 통계 수집 스크립트 — 스니펫

```bash
# secu에서 패킷 캡처 후 요청 간격(CoV) 분석
tcpdump -i any -nn -w /tmp/recon.pcap 'host 10.20.30.80' &
# ... (실험 실행) ...
# 종료 후
tshark -r /tmp/recon.pcap -Y 'http.request' -T fields \
  -e frame.time_epoch -e ip.src -e http.request.uri \
  > /tmp/recon.tsv
python3 - <<'PY'
import statistics as s, csv
rows = list(csv.reader(open('/tmp/recon.tsv'), delimiter='\t'))
t = [float(r[0]) for r in rows]
iat = [b-a for a,b in zip(t, t[1:])]
print(f"mean_iat={s.mean(iat):.3f}s  stdev={s.pstdev(iat):.3f}s  count={len(iat)+1}")
print(f"uniq_paths={len({r[2] for r in rows})}")
PY
```

## 3.5 실험 목표 지표 (학생 기준 달성)

- 방어 측 관찰 로그에서 *의심 이벤트* 2개 이상 수집

# Part 5: 탐지의 3가지 후보 전략 (30분)

## 5.1 전략 A — 행위 임계값

IAT·Burst·경로 다양성을 조합한 *행위 점수*. 단점: 합법 스캐너/벤더와 구분 어려움.

## 5.2 전략 B — 컨텍스트 일관성

세션·쿠키·리퍼러의 일관성을 본다. 사람 세션은 연속적, 에이전트는 이산적.
