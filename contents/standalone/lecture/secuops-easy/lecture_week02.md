>
> 방화벽 콘솔(`fw-gui.6v6.lab`)을 열어, **룰을 만들고 → 그 룰이 만들어내는 진짜 `nft` 명령을 보고

## 지난주 복습 (30초)

방화벽은 6v6 의 가장 바깥 문(ext↔pipe 경계)에서 **IP 와 포트** 를 보고 트래픽을 허용/차단합니다.
봉투의 겉면(주소·창구번호)만 보는, 가장 빠르고 단순한 1차 방어선입니다. 이번 주는 그 방화벽을
직접 다룹니다.

## 학습 목표

이번 주가 끝나면 여러분은 **콘솔만으로** 다음을 할 수 있어야 합니다.

1. 방화벽의 인터페이스(eth0/eth1)와 테이블(six_filter/six_nat) 구성을 읽는다.
2. 방화벽의 주요 디렉토리·설정파일·로그가 어디 있는지 안다.
3. 콘솔에서 룰을 만들어, **생성되는 `nft` 명령을 확인하고** 적용·삭제한다.
4. NAT(DNAT)로 내부 서비스를 외부에 공개한다.
5. **연결 추적(conntrack)** 으로 stateful 방화벽의 의미를 눈으로 본다.

7. 방화벽 활동을 **SIEM(Wazuh)** 으로 보낸다.
8. 침해대응 시나리오 10개 중 최소 5개를 직접 풀어 차단 룰을 작성한다.

## 2. 네트워크와 인터페이스 구성

**인터페이스** 메뉴를 엽니다. 방화벽은 두 개의 네트워크 다리를 가집니다.

| 인터페이스 | IP | 존 | 향하는 곳 |
|-----------|----|----|-----------|
| `eth0` | 10.20.30.1 | ext | 외부(공격자 10.20.30.202 가 있는 곳) |
| `eth1` | 10.20.31.1 | pipe | IPS 로 가는 통로 |

방화벽이 "문" 인 이유가 여기 있습니다. 한 발(eth0)은 바깥에, 한 발(eth1)은 안쪽 통로에 걸치고
있어서, 바깥에서 안으로 가려면 반드시 이 방화벽을 통과해야 합니다.

> **초보자 필수 개념 — 인터페이스 이름으로 룰을 거는 이유:** "eth0(바깥)으로 들어오는 트래픽만
> 막아라" 처럼, 어느 다리로 들어오고 나가는지로 정책을 세울 수 있습니다. 룰 만들기 화면의
> `iif`(들어오는 인터페이스), `oif`(나가는 인터페이스) 가 바로 이것입니다.

## 3. 주요 디렉토리 · 설정파일 · 로그파일

초보자가 꼭 알아야 할 "방화벽의 집 구조" 입니다.

| 항목 | 위치 | 설명 |
|------|------|------|
| 명령 도구 | `/usr/sbin/nft` | nftables 본체. 모든 룰을 다루는 명령 |
| 설정파일 | `/etc/nftables.conf` | 재부팅 시 자동으로 불러오는 룰 모음(`nft -f` 형식) |
| 연결추적 도구 | `/usr/sbin/conntrack` | 현재 추적 중인 연결 보기 |
| 이벤트 로그 | `/var/log/nft_edu/events.log` | 콘솔이 적용/삭제한 동작 기록(SIEM 으로 전송) |

**중요한 사실 하나 (방화벽 로그의 진실):** 컨테이너 환경에서는 nft 의 `log` 동작이 만들어내는
패킷 로그가 파일로 남지 않고 커널 메시지로 빠져나갑니다. 그래서 우리 실습에서 "내 룰이 실제로
트래픽을 막았는지" 를 확인하는 가장 확실한 방법은 **룰의 카운터(packets/bytes)** 입니다.
이건 5절에서 직접 봅니다.

> **두 가지 룰의 집:** ① `nft list ruleset` 으로 보는 **현재 메모리에 올라간 룰**(즉시 적용, 재부팅
> 시 사라짐) ② `/etc/nftables.conf` 파일에 적힌 **영구 룰**(재부팅 후에도 유지). 콘솔의 "적용" 은
> 즉시 적용입니다. 운영에서는 검증 후 설정파일에도 반영합니다.

## 4. 룰 만들기 — 콘솔이 `nft` 명령을 만들어 주는 과정

이제 핵심입니다. **룰 관리** 메뉴를 엽니다. 현재 룰셋이 체인별로 보입니다.

방화벽 룰은 세 개의 체인 중 하나에 들어갑니다.

| 체인 | 다루는 트래픽 | 비유 |
|------|--------------|------|
| `input` | 방화벽 **자신에게** 오는 트래픽 | 안내데스크에 직접 온 손님 |
| `forward` | 방화벽을 **통과해** 내부로 가는 트래픽 | 다른 호실로 가는 손님 |
| `output` | 방화벽에서 **나가는** 트래픽 | 데스크가 밖으로 보내는 것 |

### 4.1 첫 룰 만들기 — 악성 IP 차단

상황: 공격자(10.20.30.202)가 짧은 시간에 수백 건 연결을 시도합니다. 이 IP 를 통째로 막아 봅시다.

**콘솔에서:**
1. 룰 만들기 폼에서 — 체인 `forward`, 출발지(saddr) `10.20.30.202`, 동작 `drop`, 위치 `top`.
2. **① 명령 미리보기** 를 누른다. 화면에 이렇게 뜹니다.

```
nft insert rule inet six_filter forward ip saddr 10.20.30.202 counter drop
```

3. 이 명령을 읽어 봅니다.
   - `insert` = 체인 맨 위에 넣어라(= 위치 top). 가장 먼저 평가됨.
   - `inet six_filter forward` = inet 계열 six_filter 테이블의 forward 체인에
   - `ip saddr 10.20.30.202` = 출발지 IP 가 이것이면
   - `counter` = 몇 번 걸렸는지 세고
   - `drop` = 조용히 버려라
4. **② 적용** 을 누른다. 실제로 적용되고, 우측 현재 룰셋에 새 룰이 나타납니다.

> **여러분은 방금 "버튼" 이 아니라 "명령" 을 배웠습니다.** 화면 없이 터미널에서도 똑같이
> `nft insert rule ...` 을 칠 수 있게 된 것입니다. 이것이 이 특강의 목표입니다.

### 4.2 룰 평가 순서 — top vs bottom (초보자 함정)

방화벽 룰은 **위에서 아래로** 차례로 검사하고, **허용(accept)이나 차단(drop)을 만나면 거기서
끝납니다.** 그래서 순서가 중요합니다.

예를 들어 input 체인에는 이미 `icmp accept`(ping 허용)가 있습니다. 그 **아래**에 "icmp 를 세어라"
같은 룰을 넣으면, 위의 accept 에서 이미 끝나버려 아래 룰은 영영 실행되지 않습니다(카운터가 0).
그래서 **차단 룰은 보통 위(top)에** 둡니다. 콘솔의 "위치" 선택이 이 차이를 만듭니다.

### 4.3 더 똑똑한 차단 — 속도 제한(rate limit)

SSH(22번 포트) 무차별 대입 공격을 생각해 봅시다. 22번을 통째로 막으면 정상 관리자도 못 들어옵니다.
그래서 "**분당 일정 횟수를 넘는 신규 연결만**" 막습니다.

**콘솔에서:** 체인 input, 프로토콜 tcp, dport 22, ct state `new`, rate `10/minute` + "초과분만(over)"
체크, 동작 drop, 위치 top → 미리보기:

```
nft insert rule inet six_filter input tcp dport 22 ct state new limit rate over 10/minute counter drop
```

`limit rate over 10/minute` = 분당 10회를 **초과하는** 신규 연결만 매칭 → 정상 사용자는 통과하고
무차별 대입만 걸립니다. 가용성을 해치지 않는 영리한 방어입니다.

## 5. 연결 추적(conntrack)과 카운터 — stateful 방화벽

### 5.1 stateful inspection 이란

옛날의 단순 방화벽은 패킷을 하나하나 따로 봤습니다. 현대 방화벽은 **연결(흐름)을 기억** 합니다.
한 번 허용한 연결의 **응답 패킷** 은 자동으로 통과시키는 것이죠. 이것을 **stateful inspection**
이라 합니다.

우리 방화벽의 input/forward 체인 맨 위에는 이미 이런 룰이 있습니다.

```
ct state established,related accept
```

"이미 맺어진(established) 연결이거나 그와 관련된(related) 트래픽이면 통과" 라는 뜻입니다.
덕분에 우리가 외부로 보낸 요청의 응답이 매끄럽게 돌아옵니다.

### 5.2 연결을 눈으로 보기

**Stateful (연결추적)** 메뉴를 엽니다. 현재 방화벽이 기억하는 연결들이 표로 보입니다.

| proto | state | 출발 | 목적 | assured |
|-------|-------|------|------|---------|
| tcp | ESTABLISHED | 10.20.31.1:.. | 10.20.32.120:5601 | ✔ |

- `ESTABLISHED` = 양방향으로 패킷이 오간, 정상적으로 맺어진 연결
- `[ASSURED]` = 커널이 "이 연결은 확실하다" 고 표시 — 메모리에서 잘 안 지움

### 5.3 카운터 — "내 룰이 진짜 막았나?" 의 증거

**로그·활동** 메뉴를 엽니다. 룰마다 `packets`(걸린 패킷 수)가 보입니다. 4.1 에서 만든
10.20.30.202 차단 룰의 카운터가 처음엔 0 입니다. 그런데 공격자가 다시 접속을 시도하면?

```
ip saddr 10.20.30.202 counter packets 4 bytes 240 drop
```

`packets` 가 0 → 4 로 올라갑니다. **이것이 차단이 실제로 일어났다는 가장 확실한 증거** 입니다.
컨테이너에서 패킷 로그 파일이 없어도, 카운터는 거짓말을 하지 않습니다. 운영자는 이 카운터로
"어떤 룰이 얼마나 일하고 있는지" 를 파악합니다. (카운터는 "카운터 리셋" 으로 0 으로 되돌릴 수 있습니다.)

## 6. NAT — 내부 서비스를 안전하게 공개하기

### 6.1 NAT 이란

NAT(Network Address Translation)은 **주소를 바꿔치기** 하는 기술입니다. 가장 흔한 쓰임은
**DNAT(목적지 NAT)** — "방화벽의 8088 포트로 온 요청을 내부 웹서버(10.20.32.80:80)로 돌려라".
외부는 방화벽의 주소만 보고, 진짜 내부 주소는 가려집니다.

### 6.2 콘솔에서 DNAT 만들기

**NAT** 메뉴에서 — 종류 DNAT, 인터페이스 eth0, 프로토콜 tcp, 외부 포트 8088, 대상 IP
10.20.32.80, 대상 포트 80 → 미리보기:

```
nft add rule ip six_nat prerouting iifname "eth0" tcp dport 8088 counter dnat to 10.20.32.80:80
```

- `ip six_nat prerouting` = NAT 테이블의 **prerouting**(라우팅 결정 전) 체인. DNAT 은 여기서 일어납니다.
- `dnat to 10.20.32.80:80` = 목적지를 내부 웹서버로 변환

적용 후 외부에서 `방화벽:8088` 로 접속하면 내부 웹서버로 연결됩니다. **포트 포워딩** 의 기본입니다.

> **초보자 필수 — prerouting vs postrouting:** DNAT(목적지 변경)은 라우팅 *전*(prerouting),
> SNAT(출발지 변경)은 라우팅 *후*(postrouting)에 합니다. "어디로 갈지 정하기 전에 목적지를 바꾸고,
> 다 정한 뒤 출발지를 바꾼다" 고 외우면 쉽습니다.

## 7. SIEM(Wazuh) 연동

방화벽 혼자 일하면 운영자가 일일이 콘솔을 봐야 합니다. **SIEM** 은 모든 장비의 기록을 한 곳에
모아 보는 관제실(Wazuh, 10.20.32.100)입니다.

**SIEM 연동** 메뉴를 엽니다.
- 현재 상태(연동 켜짐/꺼짐, 에이전트 가동 여부, 매니저 주소)가 보입니다.
- **"연동 켜기"** 를 누르면, 방화벽의 Wazuh 에이전트가 콘솔의 이벤트 로그
  (`/var/log/nft_edu/events.log`)를 감시(tail)하도록 설정되고 에이전트가 재시작됩니다.
- 이후 콘솔에서 룰을 적용/삭제할 때마다 그 기록이 JSON 으로 이벤트 로그에 남고, Wazuh 매니저로
  전송됩니다. SIEM 대시보드에서 방화벽 활동을 시간순으로 볼 수 있습니다.

> 이렇게 각 장비를 SIEM 에 연동하는 것이 "**통합 관제**" 의 출발점입니다. W3·W5 에서 IPS·WAF 도
> 같은 방식으로 SIEM 에 연결합니다.

## 8. 방화벽 로그(활동) 분석 — 운영자의 눈

운영자는 **로그·활동** 화면에서 두 가지를 봅니다.

1. **룰별 카운터** — 어떤 룰이 트래픽을 얼마나 막고 있나. packets 가 갑자기 치솟으면 공격 신호일
   수 있습니다. 평소 0 이던 차단 룰의 카운터가 급증 → 그 출발지를 조사합니다.
2. **이벤트 로그** — 누가(콘솔) 언제 어떤 룰을 적용/삭제했나. 운영 변경 이력입니다. 침해 후
   "누가 방화벽 설정을 바꿨나" 를 추적하는 감사 자료가 됩니다.

**분석의 3단계(이번 주 습관 들이기):**
- ① **무엇이 보이나** — 어떤 룰의 카운터가 비정상인가?
- ② **무슨 의미인가** — 그 출발지/포트가 어떤 공격일 수 있나?
- ③ **무엇을 할까** — 추가 차단 룰? rate limit? SIEM 경보 확인?

## 9. 침해대응 훈련 — 시나리오 10개

**침해대응 훈련** 메뉴에는 실제 상황을 가정한 10개 시나리오가 있습니다. 각 시나리오는
① 상황 설명 ② 공격 재현 방법 ③ 만들 룰(폼 입력값) ④ 힌트 ⑤ 검증 버튼으로 구성됩니다.

## 10. 이번 주 정리

- 방화벽은 **IP·포트** 를 보는 가장 바깥 관문. 두 인터페이스(eth0 ext / eth1 pipe)로 경계를 지킨다.
- 룰은 `input`/`forward`/`output` 체인 중 하나에 들어가고, **위에서 아래로** 평가되며 accept/drop 에서
  끝난다(순서 중요).
- 콘솔의 모든 동작은 **실제 `nft` 명령** 으로 변환되어 적용된다.
- **stateful**: established/related 는 자동 통과. **conntrack** 으로 연결을 보고, **카운터** 로 차단을 증명한다.
- **NAT(DNAT)** 으로 내부 서비스를 외부에 공개(prerouting).
- 방화벽 활동을 **SIEM(Wazuh)** 으로 보내 통합 관제한다.
- 침해대응 시나리오 10개로 실전 룰 작성을 연습한다.
